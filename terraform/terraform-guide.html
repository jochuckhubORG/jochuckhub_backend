Q1. Terraform은 "인프라 생성"만, 배포는 별도 수단으로

Q2. AWS CLI vs Terraform — 관계와 차이

AWS CLITerraform
역할 AWS API를 명령어로 직접 호출 AWS 리소스를 코드로 선언하고 관리
설치 이유 AWS 인증 정보(~/.aws/credentials) 저장 및 API 호출 Terraform 명령어 사용
관계 Terraform은 내부적으로 AWS SDK를 직접 사용.
AWS CLI와는 별개지만, aws configure로 저장한
인증 정보(~/.aws/credentials)는 공유합니다.
정확히는: terraform apply를 실행하면 Terraform이 직접 AWS API를 호출합니다. AWS CLI를 거치는 게 아니라, aws configure로 저장된 인증 키 파일만 공유해서 사용합니다.

Q3. terraform/ 파일별 역할과 구성요소 분석

main.tf 핵심 파일

실제로 AWS에 만들 리소스들을 정의하는 파일. "EC2 만들어라", "Security Group 만들어라" 같은 선언을 합니다.

📄 main.tf — 블록별 설명
# ① Terraform 자체 설정
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"   # AWS 플러그인 출처
      version = "~> 5.0"          # 5.x 버전 사용
    }
  }
}

# ② 어느 AWS 리전에 만들지
provider "aws" {
  region = var.aws_region         # variables.tf의 변수 참조
}

# ③ 데이터 소스: 기존에 있는 것을 "조회"만 함 (생성 X)
data "aws_ami" "amazon_linux_2023" {
  most_recent = true              # 가장 최신 AMI
  owners      = ["amazon"]       # Amazon 공식 이미지만
  filter { ... }                  # Amazon Linux 2023 필터
}

# ④ 리소스: 실제로 AWS에 "생성"할 것들
resource "aws_security_group" "app_sg" {
  name = "jochuckhub-sg"
  ingress { from_port = 22  ... } # SSH 허용
  ingress { from_port = 8080 ... }# Spring Boot 포트 허용
  egress  { ... }                 # 아웃바운드 전체 허용
}

resource "aws_instance" "app" {
  ami           = data.aws_ami... # 위에서 조회한 AMI ID 사용
  instance_type = "t2.micro"
  key_name      = var.key_pair_name
  vpc_security_group_ids = [aws_security_group.app_sg.id]
  user_data = templatefile(       # EC2 부팅 시 실행할 스크립트
    "user_data.sh", { ... }       # 변수 주입
  )
}

variables.tf 변수 선언

"이런 변수들을 입력받겠다"고 선언하는 파일. 실제 값은 terraform.tfvars에서 넣습니다.

📄 variables.tf
variable "aws_region" {
  description = "AWS 리전"        # 설명 (선택)
  type        = string            # 타입 검증
  default     = "ap-northeast-2" # 기본값 (tfvars 없어도 됨)
}

variable "db_password" {
  description = "MySQL 비밀번호"
  type        = string
  sensitive   = true             # 로그에 값이 출력되지 않음
}
변수명설명어디서 사용?
aws_regionAWS 리전 (기본: 서울)provider 블록
key_pair_nameEC2 SSH 키페어 이름aws_instance
db_passwordMySQL 비밀번호user_data.sh로 주입
jwt_secretJWT 서명 키user_data.sh → app.env
kakao_client_id카카오 REST API 키user_data.sh → app.env
kakao_client_secret카카오 Client Secretuser_data.sh → app.env
kakao_redirect_uri카카오 OAuth 콜백 URLuser_data.sh → app.env
kakao_frontend_redirect_uri로그인 후 이동 URLuser_data.sh → app.env

terraform.tfvars 실제 값 입력

variables.tf에서 선언한 변수의 실제 값을 넣는 파일. 민감 정보가 담기므로 .gitignore에 등록되어 있습니다.

📄 terraform.tfvars (예시)
aws_region    = "ap-northeast-2"
key_pair_name = "my-key"          # AWS 콘솔에서 만든 키페어 이름

db_password         = "mypassword123"
jwt_secret          = "base64인코딩된키..."
kakao_client_id     = "카카오REST키"
kakao_client_secret = "카카오시크릿"

kakao_redirect_uri          = "http://43.201.xx.xx:8080/api/auth/kakao/callback"
kakao_frontend_redirect_uri = "http://43.201.xx.xx:8080/test-login.html"

outputs.tf 출력값

terraform apply 완료 후 터미널에 출력되는 값들. EC2 IP 등 이후 작업에 필요한 정보를 확인하기 위해 사용합니다.

📄 outputs.tf
output "ec2_public_ip" {
  value = aws_instance.app.public_ip  # 생성된 EC2의 IP
}
output "app_url" {
  value = "http://${aws_instance.app.public_ip}:8080"
}
output "ssh_command" {
  value = "ssh -i ~/.ssh/my-key.pem ec2-user@..."
}

apply 완료 시 이런 식으로 터미널에 출력됩니다:

ec2_public_ip = "43.201.xx.xx"
app_url       = "http://43.201.xx.xx:8080"
ssh_command   = "ssh -i ~/.ssh/my-key.pem ec2-user@43.201.xx.xx"

user_data.sh EC2 초기화 스크립트

EC2가 처음 부팅될 때 딱 1번 자동 실행되는 bash 스크립트. Terraform이 이 파일을 EC2에 주입합니다.

실행 단계내용
① Java 17 설치 dnf install java-17-amazon-corretto-headless
② MySQL 설치 · 시작 dnf install mysql-server → systemctl start mysqld
③ DB · 유저 생성 jochuckhub 데이터베이스, jochuck 유저 생성
④ 환경변수 파일 생성 /opt/jochuckhub/app.env — DB URL, JWT 키, 카카오 키 등
⑤ systemd 서비스 등록 /etc/systemd/system/jochuckhub.service 작성
JAR 업로드 후 systemctl start jochuckhub으로 앱 시작
참고: user_data.sh 안에 ${변수명} 형태는 Terraform이 값을 채워 넣는 템플릿 변수입니다. systemd 서비스 파일 안에 쓰인 $${ }는 실제 쉘 변수(${ })로 변환됩니다. ($$ → Terraform 템플릿에서 $ 리터럴 이스케이프)

단계명령어 / 행동어디서

값 채워 넣기		 내 PC

terraform plan

(terraform/ 폴더 안)

⑦ 앱 시작 ssh -i ~/.ssh/my-key.pem ec2-user@<IP>
